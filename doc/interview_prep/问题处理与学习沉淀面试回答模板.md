## 常见问题类别（结合项目场景）

- 需求不清晰与变更频繁：规格不完整、边界条件缺失。
- 环境与依赖：交叉编译、库版本不一致、驱动兼容问题。
- 跨平台兼容：Windows/Ubuntu/ARM 差异、字节序/路径/编码问题。
- 性能瓶颈：实时数据处理延迟高、UI 刷新卡顿、内存分配频繁。
- 并发与线程安全：数据竞争、死锁、队列拥塞、迭代器失效。
- 通信不稳定：串口/RS485 抗干扰不足、TCP 粘包拆包、MQTT 重连。
- 第三方库冲突：Qt/编译器/依赖版本差异导致行为不一致。

## 处理方法论（一步步可落地）

- 明确问题定义：复述现象与期望结果，界定范围与影响面。
- 快速复现：最小化复现场景（最小可复现程序/独立测试用例），确保可重复触发。
- 分层定位：输入→算法→通信→UI 分层排查，抓关键路径日志与指标。
- 数据驱动验证：建立度量（延迟、吞吐、CPU/内存占用、丢包率），用实验对比方案。
- 迭代修复：最小可行修复（MVP），配套回归测试与监控，逐步优化。
- 根因与防再发：撰写 Root Cause 分析，配套防护（限速、降采样、重试、超时回退）。
- 沉淀输出：模板化复盘、工具化脚本、知识库条目、代码示例与最佳实践。

## 项目化案例（可按场景选择讲）

- 实时处理卡顿（带电检测终端）：
  - 现象：采样提升后 UI 卡顿、处理延迟波动。
  - 方案：生产者-消费者队列解耦；热点路径换 SPSC 无锁环形队列；UI 刷新 40ms 控频、预分配 buffer；重计算移出 `paintEvent`。
  - 结果：延迟与 CPU 占用显著下降，刷新稳定在几十 fps。

- 串口粘包与抗干扰（设备通信）：
  - 现象：现场干扰导致错包、半包。
  - 方案：统一帧格式（帧头/长度/命令/CRC16）；接收状态机解析；轮询与超时重试；错误计数与告警。
  - 结果：成功率提升、问题定位可视化。

- TCP 服务并发（营业厅平台）：
  - 现象：连接数增多时线程-per-connection 模式资源开销大。
  - 方案：非阻塞 I/O + I/O 复用事件驱动；路由 RESTful；统一超时与心跳管理。
  - 结果：资源利用率提升，长/短连接混合场景稳定。

- 并发数据竞争（C++）：
  - 现象：偶发崩溃/数据异常。
  - 方案：RAII 锁与锁粒度优化；原子变量处理标志位；多读单写或复制-交换；迭代器失效规避（`erase(it++)`）。
  - 结果：并发稳定，问题复现与回归用例固化。

## 两三分钟综合回答（面试现场版）

> 日常工作中我遇到的问题主要集中在需求不清晰、跨平台兼容、性能瓶颈、并发与通信稳定性这些方面。我的处理方法是先**明确问题定义**，构建最小复现场景，按输入→算法→通信→UI 分层定位，并用延迟、吞吐、CPU/内存等指标做**数据驱动验证**。修复上采用最小可行方案并配回归测试与监控，随后做根因分析与**防再发**。比如在带电检测终端的实时显示里，我用生产者-消费者队列解耦，热点路径换成 SPSC 无锁环形队列，UI 控频与预分配，最终把延迟和 CPU 占用显著降下来；在设备通信里，通过统一帧格式加接收状态机与 CRC/重试，提升抗干扰与可靠性。与此同时，我会把解决过程**沉淀为模板与工具**，维护知识库与示例代码，保证团队可复用。过去我更强调广度，现在 AI 让广度补齐更容易，我更聚焦于在 C++ 并发、Qt 实时绘制与设备通信协议这些关键方向**做深度锤炼**，通过复盘与推广，把个人能力转化为团队的持续产出。
